⚠️ Pontos de Melhorias na Arquitetura do Código

🔴 1. Template não herda um layout base

Boa prática no Django:

Criar um base.html

Usar {% extends %} e {% block %}

📌 Problema atual:

HTML duplicado em várias páginas

Difícil manutenção

📌 O que falta:

{% extends "base.html" %}
{% block content %}
    <!-- formulário -->
{% endblock %}


🔴 2. Falta tratamento para CPF duplicado

Você valida e-mail, mas:

CPF normalmente também deve ser único


🔴 3. View grande demais (excesso de responsabilidade)

A função criar_cadastro:

Faz parsing

Valida tudo

Cria usuário

Retorna mensagens

📌 Boa prática no Django:
➡️ Usar Forms (forms.py)

Problema atual

Código difícil de manter

Repetição de validações no futuro

🔴 4. Validação manual repetitiva

Exemplo:

if not nome or len(nome) < 3:


📌 Django já oferece:

forms.CharField(min_length=3)

EmailField

clean_<campo>()

🔴 5. Campo CPF e Telefone sem máscara

Não é erro funcional, mas:

UX ruim

Usuário pode digitar qualquer coisa

📌 Ideal:

Usar máscara com JavaScript

Ou validar formato visualmente no frontend

🔴 6. Falta controle de sessão

No login:

# request.session['usuario_id'] = user.id


📌 Problema:

Usuário não fica autenticado de fato

Não há proteção de rotas

📌 MELHORIAS

* login(request)

Pontos pendentes / melhorias

❌ Não trata mensagens de erro via django.contrib.messages

❌ Não impede acesso de usuários já autenticados

❌ Não usa o sistema de autenticação nativo do Django

Boas práticas recomendadas

Usar messages.error() para feedback

Redirecionar usuários já logados

Preferir django.contrib.auth.authenticate

* cadastrar_usuario(request)

Pontos pendentes / melhorias

❌ View redundante com criar_cadastro

❌ Nenhuma lógica de negócio associada

❌ Pode ser fundida com a view de criação

Boas práticas recomendadas

Unificar renderização e criação em uma única view

Utilizar Form ou ModelForm


* recuperacao_senha(request)

Pontos pendentes / melhorias

❌ Não implementa fluxo real de recuperação

❌ Não envia e-mail

❌ Não gera token de redefinição

Boas práticas recomendadas

Usar PasswordResetView do Django

Configurar backend de e-mail

Nunca implementar recuperação manualmente


* acesso_painel(request)

Pontos pendentes / melhorias

❌ Não verifica se o usuário está autenticado

❌ Permite acesso direto por URL

Boas práticas recomendadas

Usar @login_required

Validar sessão ou autenticação


* historico(request)

Pontos pendentes / melhorias

❌ Não filtra dados por usuário

❌ Página acessível sem autenticação

Boas práticas recomendadas

Restringir acesso por usuário logado

Buscar dados relacionados ao usuário autenticado


* criar_cadastro(request)

Pontos críticos a corrigir

❌ Não valida campos obrigatórios

❌ Não trata CPF duplicado

❌ Não trata e-mail duplicado

❌ Não valida formato dos dados

❌ Mistura POST e GET na mesma lógica

❌ Expõe todos os usuários no GET (UserCadastro.objects.all())

Boas práticas recomendadas

Usar ModelForm

Validar dados antes de salvar

Remover listagem de usuários dessa view

Tratar exceções de integridade (IntegrityError)

Separar cadastro e listagem


* verificar_login(request)

Pontos críticos

❌ Reimplementa sistema de autenticação do Django

❌ Sessão comentada e incompleta

❌ Não usa login() e logout()

❌ Redirecionamento inconsistente

❌ Falta proteção contra brute force

Boas práticas recomendadas

Usar authenticate() e login()

Usar django.contrib.auth.models.User

Centralizar autenticação no sistema nativo

📌 Verificação


### ❌ Campos obrigatórios não verificados

**Problema**

- Se um campo vier vazio, o registro ainda é criado.

**O que falta validar**

- Campos nulos ou vazios antes do `.create()`.

---

### ❌ Unicidade de dados

**Problema**

- Nenhuma verificação para:
    - CPF duplicado
    - E-mail duplicado

**Impacto**

- Usuários duplicados no banco.

---

### ❌ Integridade da senha

**Problema**

- A senha é hasheada, mas:
    - Pode ser vazia
    - Pode ser curta
    - Pode ser igual ao CPF ou e-mail
    

🔐 RN-06 — Política de senha

Regras

Senha deve conter:

Mínimo de caracteres

Letras e números

Senha não pode:

Ser igual ao CPF

Ser igual ao e-mail

Justificativa
Segurança básica de credenciais.

---

### ❌ Validação de tipo de dado

**Problema**

- Todos os valores chegam como string.
- Não há conversão ou validação semântica.

**Exemplos**

- `data_nasc` não validada como `date`
- CPF e telefone não tratados como números

---

OBS

### ❌ Exposição indevida de dados

**Problema**

```python
teste = UserCadastro.objects.all()

```

**Risco**

- Retorna todos os usuários sem filtro.
- Quebra LGPD e segurança.

**O que falta validar**

- Se o usuário pode visualizar esses dados (provavelmente não).

### ❌ Fluxo de autenticação incompleto

**Problemas**

- Não verifica se o usuário está ativo
- Não valida status da conta
- Não valida tentativas excessivas

### ❌ Sessão não validada

**Problema**

- Sessão comentada
- Nenhum controle de acesso persistente

### ❌ Retorno inconsistente

**Problema**

```python
return render(request,'home/index.html')

```

**Risco**

- Usuário pode acessar página sem login.

## 4. Validações ausentes em views de renderização simples

### `login`, `cadastrar_usuario`, `recuperacao_senha`

**Problema**

- Nenhuma verificação de contexto:
    - Usuário já autenticado
    - Sessão ativa
